# Patent Emerging Radar — Full Project Notes

## The Core Idea

Use the USPTO patent database (1976–2025, 9.4M patents) to **discover and forecast
emerging research topics from scratch** — without relying on a pre-curated list.

Patents are filed **1–3 years before academic papers** are published. A researcher
files a patent the moment they have something novel and protectable — long before
peer review completes. By clustering patent titles semantically and tracking those
clusters over time, we can spot the next CRISPR or LLMs before most people notice.

## Why This Is Better Than the Original Project

The `scientific-trend-forecaster` project (sibling folder) tracks 37 hand-curated
topics using publication data. This project goes further:

| scientific-trend-forecaster | patent-emerging-radar |

## Pipeline

### Stage 1: Embed (scripts/embed_patents.py)
- Streams `g_patent.tsv.zip`, filters to granted utility patents (wipo_kind B2/B1)
- Encodes titles with `all-MiniLM-L6-v2` (384-d, normalised)
- Saves one `.npz` per year: `ids`, `dates`, `titles`, `embeddings`
- **CPU time:** ~8–12 min/year. Run `--years 1996 2024` overnight (~4–6 hrs)
- **Kill-safe:** completed years are skipped on re-run
- **GPU time:** ~1–2 min/year

### Stage 2: Cluster (notebook Part 3)
- Load embeddings for rolling 2-year windows
- UMAP: 384-d → 10-d (cosine metric, 50 neighbours)
- HDBSCAN: density-based, no fixed K, marks noise as -1
- Cross-window stitching: cosine sim ≥ 0.85 → same cluster identity
- Outputs: cluster time series, centroids, sample titles

### Stage 3: Feature Engineering (notebook Part 4, 18 features)
| Feature | Description |
|---|---|
| `log_count` | log(count+1) |
| `count_share` | cluster ÷ global quarter total |
| `velocity` | QoQ Δlog_count |
| `acceleration` | Δvelocity |
| `jerk` | Δacceleration |
| `mom_4q` | YoY log-count change |
| `mom_8q` | 2-year log-count change |
| `roll_mean_4q` | 4Q rolling mean |
| `roll_std_4q` | 4Q rolling std (volatility) |
| `roll_mean_8q` | 8Q rolling mean (trend baseline) |
| `roll_std_8q` | 8Q rolling std |
| `above_trend` | log_count − roll_mean_8q |
| `macd` | roll_mean_4q − roll_mean_8q |
| `z_score` | above_trend / (roll_std_8q + ε) |
| `global_rank_pctl` | percentile rank among all clusters, same quarter |
| `log_cumsum` | log(cumulative count) — maturity signal |
| `sin_q`, `cos_q` | cyclical quarter encoding |

### Stage 4: LSTM + GRU (notebook Parts 5–6)
- Input: 8-quarter window (2 years) × 18 features
- Target: log_count 8 quarters ahead (2-year forecast)
- Loss: Huber(delta=1.0) — robust to count spikes
- Primary metric: Spearman rank correlation (ranking matters more than exact values)
- Both GRU and LSTM trained, compared on val set

### Train / Val / Test Split (anchor-point based, no leakage)
| Split | Anchor range | Purpose |
|---|---|---|
| Train | 2005 Q1 → 2017 Q4 | Fit weights |
| Val | 2018 Q1 → 2020 Q4 | Tune hyperparameters |
| Test | 2021 Q1 → 2023 Q4 | **LOCKED — one shot** |
| Future | 2024 Q1 → present | Leaderboard inference |

**Test gate:** `RUN_TEST_SET = False` in notebook Part 5. Raises `RuntimeError` until unlocked.

## Validation Strategy

### Why the original 37-topic panel is bad for validation
The `scientific-trend-forecaster` panel.csv was built from 37 hand-picked topics chosen
*because they were already known to be important*. Using it as validation is circular.

### What we use instead
**62 objective eruption events** from `eruption_ground_truth.csv`, derived purely from
the OpenAlex 4,516-topic panel with no manual selection:

- **Type A (emerging):** tiny topic (<500 papers/yr) that grew 5× in 5 years
- **Type B (acceleration):** established field (<10k papers/yr) that surged 4× in 5 years

**Nobel Prize cross-check (Layer 2):**
- `T11289` Single-cell/spatial transcriptomics → confirmed eruption 2014→2019 (4.5×), Nobel 2024
- CRISPR, gravitational waves, cancer immunotherapy are confirmed events but grow more gradually
  (2–3× over 5 years) — they fall outside the eruption thresholds but serve as qualitative checks

### The validation metric
For each eruption event, find the matching patent cluster, find when it first "took off"
(velocity ≥ 0.3), compute:
```
lead_quarters = (eruption_year − takeoff_year) × 4
```
Positive = patent cluster grew BEFORE the publication eruption. This is the headline result.

## Patent Data — Year Range Decision

The TSV goes back to 1976 (9.4M patents total):

| Era | Patents/yr | Value | Recommend |
|---|---|---|---|
| 1996–2024 | 120k–390k | Internet/genomics era, modern language | **Yes — default** |
| 1990–1995 | 100–115k | Early biotech, older language | Maybe |
| 1976–1989 | 60–90k | Pre-internet, limited relevance | No |

Default in `embed_patents.py`: `--years 1996 2024`

The stream_patents filter is fully parameterised — just pass different `--years` to go further back.
